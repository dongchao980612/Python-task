Introduction
简介

Teeko is a 2 player strategy board game invented by John Scarne in 1937. In this game, players first take turns placing pieces on a 5x5 game board. Players then continue to take turns moving these pieces with the goal of being the first player to line their pieces up in either an unbroken line (horizontal, vertical, or diagonal) or a perfect square. This website gives a demonstration of the basic rules. You will be implementing a text based version of Teeko in this assignment. Note that this document takes precedence if there is any difference between it and the behavior of the above website. Instead of using colors, in our text based version of the game players will be represented using special symbols.
Teeko 是由约翰-斯卡纳（John Scarne）于 1937 年发明的双人策略棋盘游戏。在这款游戏中，玩家首先要轮流在 5x5 游戏板上放置棋子。然后，玩家继续轮流移动这些棋子，目标是成为第一个将棋子排成一条不间断线（水平线、垂直线或对角线）或一个完美正方形的玩家。该网站演示了基本规则。在本作业中，您将执行基于文本的 Teeko 版本。请注意，如果本文档与上述网站的行为有任何差异，则以本文档为准。在我们的文字版游戏中，玩家将使用特殊符号来表示，而不是使用颜色。

Gameplay takes place on a square board partitioned into positions. Players can place pieces on the board at empty positions, and move these pieces from one position to an adjacent empty position. Adjacent positions can either be horizontal, vertical, or diagonal. Players cannot place or move their opponent’s pieces. Player one’s pieces are represented using the special symbol ◍, and player two’s pieces are represented using the special symbol ●. These symbols are provided as constants in support.py and may appear slightly different on different operating systems such as Mac or Windows. Each Player has a total of four pieces. On a given player’s turn, they are not allowed to move any pieces unless all four of their pieces are already on the board. Players also cannot place another piece on the board on their turn if all four pieces are already placed. The game ends when one player has arranged their pieces in either an unbroken line (horizontal, vertical, or diagonal), or a perfect square. When this happens, the player that managed to arrange their pieces in this way is declared the winner.
游戏在方形棋盘上进行，棋盘被分割成多个位置。玩家可以在棋盘上的空位置放置棋子，并将这些棋子从一个位置移动到相邻的空位置。相邻位置可以是水平、垂直或对角线。玩家不能放置或移动对手的棋子。一号棋手的棋子用特殊符号 ◍ 表示，二号棋手的棋子用特殊符号 ● 表示。这些符号在 support.py 中作为常量提供，在 Mac 或 Windows 等不同操作系统上可能略有不同。每个玩家共有四个棋子。在某一玩家的回合中，除非他们的四个棋子都已在棋盘上，否则不允许移动任何棋子。如果四个棋子都已摆好，玩家也不能在自己的回合中在棋盘上摆放另一个棋子。当一方将棋子排成一条不间断的直线（水平线、垂直线或对角线）或一个完美的正方形时，游戏结束。当出现这种情况时，成功摆放棋子的玩家即宣布获胜。

The current game state is displayed.
显示当前游戏状态。

显示一条信息，说明轮到谁了，以及是否需要放置或移动棋子。如果棋手仍有棋子需要放置，还会显示剩余棋子的数量。

The player is prompted to enter a command.

The game state is checked. If a player has won, the state of the game and a message are displayed announcing their victory, and the game immediately proceeds to Step 7.
检查游戏状态。如果玩家获胜，则会显示游戏状态和宣布获胜的信息，游戏立即进入步骤 7。

Player turn alternates and gameplay returns to step 1.

Code Style
代码风格

The style of your assignment will be assessed by a tutor. Style will be marked according to the style rubric provided with the assignment. The style mark will be out of 40.

You are expected to follow the PEP-8 style guidelines discussed in lectures. The key consideration in marking your code style is whether the code is easy to understand. There are several aspects of code style that contribute to how easy it is to understand code. In this assignment, your code style will be assessed against the following criteria:
您应遵循讲座中讨论的 PEP-8 风格指南。代码风格评分的主要考虑因素是代码是否易于理解。代码风格的几个方面会影响代码的易懂程度。在本次作业中，将根据以下标准对您的代码风格进行评估：

Readability
可读性

Program Structure: Layout of code makes it easy to read and follow its logic. This includes using whitespace to highlight blocks of logic, and ensuring all lines are below 80 characters.
程序结构：代码布局使其易于阅读并遵循其逻辑。这包括使用空白来突出逻辑块，并确保所有行少于 80 个字符。

Descriptive Identifier Names: Variable, constant, and function names clearly describe what they represent in the program’s logic. Do not use Hungarian Notation for identifiers. In short, this means do not include the identifier’s type in its name, rather make the name meaningful (e.g. employee identifier).
描述性标识符名称：变量名、常量名和函数名应清楚地描述它们在程序逻辑中的含义。标识符不要使用匈牙利符号。简而言之，这意味着不要在名称中包含标识符的类型，而是要让名称更有意义（如雇员标识符）。

Named Constants: Any non-trivial fixed value (literal constant) in the code is represented by a descriptive named constant (identifier).
已命名常量：代码中任何非三维固定值（字面常量）都用一个描述性命名常量（标识符）来表示。

Algorithmic Logic
算法逻辑

Single Instance of Logic: Blocks of code should not be duplicated in your program. Any code that needs to be used multiple times should be implemented as a function.
逻辑的单一实例：程序中的代码块不应重复。任何需要多次使用的代码都应作为函数来实现。

Variable Scope: Variables should be declared locally in the function in which they are needed. Global variables should not be used.
变量范围：应在需要使用变量的函数中本地声明变量。不应使用全局变量。

Control Structures: Logic is structured simply and clearly through good use of control structures (e.g. loops and conditional statements).
控制结构：充分利用控制结构（如循环和条件语句），使逻辑结构简单明了。

Documentation:
文档

Comment Clarity: Comments provide meaningful descriptions of the code. They should not repeat what is already obvious by reading the code (e.g. # Setting variable to 0). Comments should not be verbose or excessive, as this can make it difficult to follow the code. Comments must be written in English.
注释清晰：注释对代码进行有意义的描述。注释不应重复阅读代码就能明显看出的内容（如 # 将变量设置为 0）。注释不应冗长或过多，否则会导致难以理解代码。注释必须用英语书写。

Informative Docstrings: Every function should have a docstring that summarises its purpose. This includes describing parameters and return values (including type information) so that others can understand how to use the function correctly.
内容丰富的文档说明：每个函数都应有一个 docstring，概述其目的。这包括描述参数和返回值（包括类型信息），以便他人了解如何正确使用该函数。

Description of Logic: All significant blocks of code should have a comment (in English) to explain how the logic works. For a small function, this would usually be the docstring. For long or complex functions, there may be different blocks of code in the function. Each of these should have an in-line comment describing the logic.
逻辑说明：所有重要的代码块都应该有注释（英文）来解释逻辑是如何工作的。对于小型函数，这通常就是 docstring。对于长函数或复杂函数，函数中可能有不同的代码块。每个代码块都应有一个行内注释来说明逻辑。
